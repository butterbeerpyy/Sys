# 7.2日理解
## 阶段一 cpu直写模式
工作流程是极其“原始”且依赖 CPU 的：

CPU 喂数据（写）： CPU 执行 STORE 指令，把矩阵 A 和矩阵 B 的数据，一个字一个字地从系统里硬塞进 VMAC 内部的寄存器（也就是你报告里的 reg_a 和 reg_b）。

CPU 发指令（启）： 数据填满后，CPU 往 VMAC 的控制寄存器（ctrl_reg）里写一个特定的值（比如写个 1），相当于按下了工厂机器的“启动开关”。

VMAC 闪电计算： VMAC 的硬件电路瞬间完成乘加运算，把结果锁存在结果寄存器（reg_c）里。

CPU 取结果（读）： CPU 再执行一堆 LOAD 指令，把 reg_c 里的结果一个字一个字地读出来，存回到系统的内存（RAM）里。

直写模式的痛点： 虽然算得极快，但 CPU 把大量时间全花在“把数据搬进搬出 VMAC”这件苦力活上了。这也是为什么后面必须做 DMA 优化的原因。

## 为什么需要 VMAC？

假设我们要算一个最简单的 $2 \times 2$ 矩阵乘法。如果只用 CPU（纯软件算）： CPU 执行的是指令集（比如 RISC-V 指令）。它得跑一个嵌套的 for 循环。从内存读 A，读 B，做乘法，做加法，再写回内存。如果加上 VMAC（硬件加速）： VMAC 里面是用 Verilog 直接连好的纯硬件逻辑电路（一堆乘法器和加法器）。数据只要一喂进去，电流一过，一个时钟周期（或者几个周期）就能把整个矩阵的乘加结果全部“吐”出来。这叫空间换时间。PicoRV32 留了接口（比如内存映射 I/O，MMIO），就是为了让你把这个自己捏的“硬件计算器”挂到系统总线上去。

## 工作量
CPU 直写模式测试（8 轮随机：

地址解码逻辑（Address Decoding）：
在系统总线上，内存（RAM）有自己的地址（比如 0x0000 开始），VMAC 也有自己的地址范围。代码里必须写一段逻辑：当 CPU 访问 0x2000 之类的特定地址时，要把数据精准地拦下来，存进 VMAC 的配置寄存器或数据寄存器，而不是丢进 RAM。

矩阵计算核心（Compute Core）：
用 Verilog 实现了真正的硬件矩阵乘法器。这不是写个软件代码 A * B 那么简单，而是要在硬件层面描述乘法器阵列是怎么连线的，怎么把 reg_a 和 reg_b 的线牵出来做乘加运算，然后送给 reg_c。

接口握手（Bus Interface）：
当 CPU 通过总线把数据扔过来的时候，VMAC 需要按照总线协议（比如 valid 信号有效、回复 ready 信号）正确地接收数据，不漏接、不卡死。

先回答你的三个核心问题：

1. **这是经典五级流水线（IF-ID-EX-MEM-WB）吗？** —— **不是。**
2. **CPU 直写模式的时候有实现流水吗？** —— **绝对没有，纯串行。**
3. **流水都是在哪里实现的？** —— **在最后阶段的 DMA 控制器状态机里（Task 4）。**

下面我们结合你刚考完的体系结构知识，把这几个点剥洋葱一样讲清楚：

### 1. 你的流水线 vs 经典五级流水线

你在课上学的经典五级流水（取指、译码、执行、访存、写回）叫做 **指令流水线（Instruction Pipeline）**，它的目的是让 CPU 每一拍都能执行一条新指令。

而你在这个作业里实现的，叫做 **数据流水线 / 任务流水线（Data/Task Pipeline）**。
你的 VMAC 硬件加速器不负责取指令和译码（这是 PicoRV32 那个 CPU 核心干的事），你的加速器只干两件事：**搬数据（Load/Store）** 和 **算乘加（Compute）**。
你实现的流水线，是在“批次（Batch）”这个维度上展开的。

### 2. 为什么 CPU 直写模式没有流水？

回想一下体系结构里的 **结构冒险（Structural Hazard）**：当两个部件抢夺同一个硬件资源时，就会发生冲突。

在 CPU 直写模式下，整个系统的流程是这样的：

1. CPU 跑 `STORE` 指令（访存），把数据塞进 VMAC。
2. CPU 发启动信号。
3. VMAC 的组合逻辑（或者多周期时序逻辑）开始算矩阵（执行）。
4. CPU 跑 `LOAD` 指令（访存），把结果读回去。

在这个阶段，**没有任何交叠执行**。因为数据全靠 CPU 的 Load/Store 单元来搬。CPU 搬砖的时候，VMAC 的乘法器在摸鱼；VMAC 算乘法的时候，CPU 只能 Stall（阻塞）在那里等它算完。这就是最典型的纯串行，效率极其拉胯。

---

### 3. 流水线到底在哪里实现的？（硬核拆解）

你作业的流水线，是在最后的 **阶段 4：DMA 流水线优化** 里，通过修改 `rtl/periph/vmac_unit.v` 里的代码实现的。

你可以把它完美类比成体系结构里的思想：为了解决 Load 单元和 Compute 单元的结构冒险，你引入了 **双缓冲（Double Buffering）** 机制。

* **原先的痛点：** 只有一个寄存器 `reg_a`。DMA 在往 `reg_a` 里写数据时，乘法器绝对不能去读它（典型的 **写后读 RAW 数据冒险**），所以必须等 DMA 写完，乘法器才能动。
* **你的天才解法：双缓冲（影子寄存器）**
你加了一组影子寄存器 `reg_a_shadow`。
现在，你的流水线被分成了两个阶段：**DMA访存（MEM）** 和 **矩阵计算（EX）**。

**它是这样跑起来的（请代入流水线的时空图）：**

* **时钟周期 T1：** DMA 把 Batch 0 的数据加载到主寄存器 `reg_a`。（仅 MEM 工作）
* **时钟周期 T2：** 乘法器开始读取 `reg_a` 计算 Batch 0 的结果。（EX 工作）
**🔥 重点来了：在同一个 T2 周期里**，DMA 并没有闲着！DMA 状态机跳转到了你新写的 `DMA_LOAD_A_PIPE` 状态，它把系统总线利用起来，开始把 Batch 1 的数据悄悄加载到影子寄存器 `reg_a_shadow` 里。（MEM 也在工作！）

看到了吗？**VMAC 的计算单元（EX）在处理第 N 个 Batch，而 DMA 控制器（MEM）在同时预加载第 N+1 个 Batch。**

在你的代码里，你是通过一个叫 `buffer_select` 的标志位（类似一个多路选择器 MUX）来切换的。这相当于在硬件层面上彻底消除了访存和计算的结构冲突，让这两个部件 100% 并行跑了起来，所以你的加速比才能飙升到 3.59 倍。

一句话总结：**你用双缓冲（Double Buffering）在硬件加速器内部，实现了一个完美的“Load - Compute”两级数据流水线！** 这在体系结构课设里绝对是拿高分的标准操作。这么讲是不是瞬间亲切多了？
