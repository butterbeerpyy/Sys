### 🐛 已修复的 Bug
**DMA 地址/数据不同步问题**（[vmac_unit.v:221-250](rtl/periph/vmac_unit.v)）

**根因**：
- 在 `DMA_LOAD_A/B` 状态中，地址计算使用 `dma_index`
- 当 `dma_ready=1` 时，同周期内递增 `dma_index`
- 但组合逻辑 `dma_addr` 还没更新，导致下一周期使用旧地址

**症状**：
```
idx=0 addr=0x500  ✓ 正确
idx=1 addr=0x500  ✗ 重复！应该是 0x504
idx=2 addr=0x504  ✓ 正确但晚了一步
```
结果：`reg_a[0:7] = [1,1,2,3,4,5,6,7]`（第一个元素重复，最后一个丢失）

**修复方案**：
使用 `dma_index - 1` 来存储数据，因为地址已经提前一个周期发出：
```verilog
if (dma_ready) begin
    if (dma_index > 0) begin
        reg_a[dma_index - 1] <= dma_rdata;  // 存储上一周期请求的数据
    end
    dma_index <= dma_index + 1;  // 递增索引用于下一地址
end
```

### 实现方案

**系统架构**：
```

    ↓

    ↓

    ↓
```


### 核心模块

#### 1. 粗扫描器 (simple_improved_scanner.v)

**功能**：计算每个 patch 的兴趣度（基于位置先验）

**状态机**：
- `IDLE` → 等待启动
- `SCAN` → 接收 12544 个像素（112×112）
- `COMPUTE` → 计算 64 个兴趣度值
- `DONE` → 输出完成信号

**兴趣度算法**（纯位置权重）：
```verilog
// 曼哈顿距离到中心
dist = |row - 4| + |col - 4|;

// 权重分配（中心优先假设）
if (dist == 0) interest = 255;      // 正中心 4 个
else if (dist == 1) interest = 200; // 周围 8 个
else if (dist == 2) interest = 150; // 再外 12 个
else if (dist == 3) interest = 100; // 更外 12 个
else interest = 0;                  // 边缘 28 个
```

**设计理由**：
- VLM 任务中主要物体通常在图像中心
- 纯位置权重避免复杂的特征提取
- 硬件友好，固定延迟

#### 2. Top-K 选择器 (multicycle_topk_selector.v)

**功能**：从 64 个值中选出最大的 16 个（不排序）

**算法**：简化的选择算法
```verilog
// 每个周期处理一个值
for each value:
    找出当前 Top-16 中的最小值
    if 新值 > 最小值:
        替换最小值
```

**性能**：
- 64 个周期完成
- 每个周期只做比较和替换
- 避免了复杂的排序逻辑

**测试用例**：狗的照片（112×112）


- 硬件延迟：~200 个时钟周期

### 技术挑战与解决

#### 挑战 1：Top-K 选择器逻辑 bug
**问题**：单周期嵌套循环导致逻辑冲突，输出始终是 `[0,1,2,3...]`  
**解决**：改成多周期版本，每周期只处理一个值

#### 挑战 2：组合逻辑时序问题
**问题**：`always @(*)` 在 SCAN 阶段就计算，此时数据无效，输出全 0  
**解决**：改成时序逻辑，在 `COMPUTE` 状态计算

#### 挑战 3：图像大小不匹配
**问题**：testbench 期望 112×112，但 Python 生成的是 32×32  
**解决**：修复 `img_to_hex.py` 参数传递

#### 挑战 4：边缘梯度主导
**问题**：即使加入位置权重，树的边缘梯度仍然太强  
**解决**：改成纯位置权重，完全忽略图像内容

## 🚀 任务 4：DMA 流水线优化（2026/06/25）

### 背景与动机

**当前问题**：
- DMA虽然减少了CPU干预，但硬件资源利用率低
- 各阶段串行执行：LOAD_A → LOAD_B → COMPUTE → STORE_C
- 性能提升有限（平均加速比仅2.81×）
- 老师要求实现流水线架构

**优化目标**：
- 实现真正的流水线执行
- 加载下一batch的同时计算当前batch
- 预期性能提升：50%+（加速比从2.81× → 4.2×）

### 技术方案

#### 核心思路：双缓冲 + 状态重叠

**当前架构（串行）**：
```
时刻    | VMAC状态
--------|----------------------------------
t0-t10  | LOAD_A(batch0)  [DMA忙, 计算空闲]
t11-t20 | LOAD_B(batch0)  [DMA忙, 计算空闲]
t21-t50 | COMPUTE(batch0) [DMA空闲, 计算忙]
t51-t60 | STORE_C(batch0) [DMA忙, 计算空闲]
t61-t70 | LOAD_A(batch1)  [重复...]
```
**资源利用率**: ~33%（DMA和计算单元大部分时间互斥）

**流水线架构（并行）**：
```
时刻    | 缓冲区0              | 缓冲区1              | 计算单元
--------|---------------------|---------------------|-------------------
t0-t10  | LOAD_A(batch0)      | -                   | -
t11-t20 | LOAD_B(batch0)      | -                   | -
t21-t50 | STORE_C(batch0)     | LOAD_A(batch1)      | COMPUTE(batch0)
t51-t80 | -                   | LOAD_B(batch1)      | COMPUTE(batch0继续)
t81-t110| LOAD_A(batch2)      | STORE_C(batch1)     | COMPUTE(batch1)
t111-.. | LOAD_B(batch2)      | -                   | COMPUTE(batch1继续)
```
**资源利用率**: ~75%+（DMA和计算并行）

#### 关键设计要点

1. **双缓冲寄存器组**
   ```verilog
   // 原有单组寄存器
   reg [31:0] reg_a [0:MATRIX_SIZE-1];
   reg [31:0] reg_b [0:MATRIX_SIZE-1];
   reg [31:0] reg_c [0:MATRIX_SIZE-1];
   
   // 新增第二组（用于预加载）
   reg [31:0] reg_a_shadow [0:MATRIX_SIZE-1];
   reg [31:0] reg_b_shadow [0:MATRIX_SIZE-1];
   reg buffer_select; // 0=使用reg_a/b, 1=使用shadow
   ```

2. **状态机扩展**
   ```verilog
   // 新增状态
   localparam [2:0] DMA_LOAD_A_PIPE   = 3'd7; // 流水加载A
   localparam [2:0] DMA_LOAD_B_PIPE   = 3'd8; // 流水加载B
   localparam [2:0] DMA_STORE_C_PIPE  = 3'd9; // 流水存储C
   
   // 状态转移逻辑
   DMA_COMPUTE: begin
       if (compute_busy) begin
           // 计算未完成，启动下一batch的加载
           if (next_batch_available)
               dma_state <= DMA_LOAD_A_PIPE;
       end else begin
           dma_state <= DMA_STORE_C;
       end
   end
   ```

3. **计算状态标志**
   ```verilog
   reg compute_busy;        // 计算单元忙标志
   reg [7:0] compute_cycle; // 计算周期计数
   
   always @(posedge clk) begin
       if (start_compute) begin
           compute_busy <= 1'b1;
           compute_cycle <= 8'd0;
       end else if (compute_busy) begin
           compute_cycle <= compute_cycle + 1;
           if (compute_cycle >= COMPUTE_LATENCY)
               compute_busy <= 1'b0;
       end
   end
   ```

4. **DMA仲裁增强**
   - 当前batch存储 vs 下一batch加载：优先存储（保证数据完整性）
   - VMAC vs VLM：保持VMAC优先

```verilog
    if (!compute_started) begin
        // 第一次进入，启动计算
        compute_busy <= 1'b1;
        compute_cycle <= 8'd0;
        compute_started <= 1'b1;

        // 启动流水加载（使用shadow缓冲）
        dma_state <= DMA_LOAD_A_PIPE;
        buffer_select <= ~buffer_select; // 切换到shadow
    end else if (!compute_busy) begin
        // 计算完成，进入存储
        dma_state <= DMA_STORE_C;
        compute_started <= 1'b0;
    end
end

DMA_LOAD_A_PIPE: begin
    // 加载到shadow寄存器，不影响当前计算
    if (dma_valid && dma_ready) begin
        if (buffer_select)
            reg_a_shadow[dma_index - 1] <= dma_rdata;
        else
            reg_a[dma_index - 1] <= dma_rdata;
        // ...
    end
    if (dma_index >= dma_limit)
        dma_state <= DMA_LOAD_B_PIPE;
end
```

### 2026/06/25 - Bug修复：状态机死锁 ✅

**问题诊断**：
- 测试超时，波形文件2GB，说明状态机陷入死循环
- 根因：在`DMA_COMPUTE`状态中间跳转到`DMA_LOAD_A_PIPE`，导致`compute_busy`永远无法清零

**修复方案**：
- 简化`DMA_COMPUTE`逻辑：计算完成后立即判断是否需要预加载
- 如果有下一batch，进入`DMA_LOAD_A_PIPE`预加载
- 如果没有下一batch，直接进入`DMA_STORE_C`
- 预加载使用`(batch_idx + 1)`的地址
- `batch_idx`在`DMA_STORE_C`完成后才递增
