## 四、数据库事务——实践练习

### 1.1 事务语法

事务是数据库操作的基本单位，保证一组操作要么全部执行，要么全部不执行。确保数据的一致性和完整性，在并发或故障时维护数据正确。

| 语法 | 含义 |
| --- | --- |
| `START TRANSACTION / BEGIN` | 表示事务开始执行。 |
| `COMMIT` | 表示事务完成所有数据操作，并且保存操作结果，标志着事务的成功完成。 |
| `ROLLBACK / END` | 表示事务未完成所有数据操作，重新返回到事务开始，标志着事务的撤销。 |

> 注：这里把 `ROLLBACK / END` 放在一起，表达的是“结束事务并撤销未提交的修改”这一类含义；在下面的示例里，实际用到的是 `ROLLBACK`。

---

### 1.2 事务示例

**例 1**：启动事务，使用 `SELECT` 语句查询 `Person` 表内容，再使用 `INSERT` 语句插入新数据，最后提交事务，接着再次查询 `Person` 表内容。

```sql
-- 创建 Person 表
CREATE TABLE Person (
    id INT PRIMARY KEY,
    name VARCHAR(255)
);
```

```sql
-- 启动事务
BEGIN;

-- 查询 Person 表内容
SELECT * FROM Person;

-- 插入新数据
INSERT INTO Person (id, name) VALUES (1, 'Alice');

-- 提交事务
COMMIT;

-- 再次查询 person 表内容
SELECT * FROM Person;
```

插入新数据之前的 `person` 表内容如下图所示：

```text
db1=# db1=#  id | name
---+------
(0 rows)
```

插入新数据并提交了事务的 `person` 表内容如下图所示：

```text
db1=# SELECT * FROM Person;
id | name
---+-------
 1 | Alice
(1 row)
```

> 注：示例里 `Person` 和 `person` 混着写。很多数据库对未加引号的标识符会按统一规则处理，但为了少踩坑，实际写 SQL 时最好保持同一种写法。

---

**例 2**：启动事务，再使用 `SELECT` 语句查询 `person` 表内容，接着插入 `id` 为 `122` 的新数据，使用 `ROLLBACK` 语句进行回滚。最后再次查询 `person` 表内容。

```sql
-- 启动事务
BEGIN;

-- 查询 person 表内容
SELECT * FROM Person;

-- 插入 id 为 122 的新数据（name 列可空）
INSERT INTO Person (id) VALUES (122);

-- 回滚事务
ROLLBACK;

-- 再次查询 person 表内容
SELECT * FROM Person;
```

插入新数据前的 `person` 表内容如图所示：

```text
db1=# SELECT * FROM Person;
id | name
---+------
 1 | Alice
(1 row)
```

回滚事务之后，再次查询 `person` 表内容，发现与事务执行前的结果相同：

```text
db1=# INSERT INTO Person (id) VALUES (122);
INSERT 0 1
db1=# ROLLBACK;
ROLLBACK
db1=# SELECT * FROM Person;
id | name
---+------
 1 | Alice
(1 row)
```

> 注：`INSERT INTO Person (id) VALUES (122);` 只写了 `id`，说明这个表的 `name` 字段允许为空，或者默认值可以为空；否则这条语句会报错。

### 1.3 事务保存点语法

事务保存点可以提供细粒度的回滚控制，设置保存点后，可针对事务内的局部错误进行调整，继续执行后续操作，减少事务开销。

| 语法 | 含义 |
| --- | --- |
| `SAVEPOINT savepoint_name` | 在当前事务里建立新的保存点，`savepoint_name` 表示保存点的名称。 |
| `ROLLBACK TO SAVEPOINT savepoint_name` | 事务回滚到保存点 `savepoint_name`。 |
| `RELEASE SAVEPOINT` | 删除一个保存点。 |

> 注：保存点的作用是“只撤回一部分操作”，不用把整个事务都撤掉；这个特性在你改了很多步、但只想撤销中间某一步时很有用。

---

### 1.4 事务保存点示例

**例 1**：开启事务，对 `person` 表进行查询，然后插入新数据，建立保存点 `s1`，然后再插入新数据，建立保存点 `s2`，然后回滚到保存点 `s1`。最后提交事务，再次查询 `person` 表内容。

```sql
-- 开启事务
BEGIN;

-- 第一次查询 person 表
SELECT * FROM person;

-- 插入新数据（假设插入 id 为 3, name 为 'InsertedBeforeS1'）
INSERT INTO person (id, name) VALUES (3, 'InsertedBeforeS1');

-- 建立保存点 s1
SAVEPOINT s1;

-- 再次插入新数据（假设插入 id 为 4, name 为 'InsertedAfterS1'）
INSERT INTO person (id, name) VALUES (4, 'InsertedAfterS1');

-- 建立保存点 s2
SAVEPOINT s2;

-- 回滚到保存点 s1（撤销保存点 s1 之后的插入操作）
ROLLBACK TO SAVEPOINT s1;

-- 提交事务
COMMIT;

-- 再次查询 person 表
SELECT * FROM person;
```

第一次查询 `person` 表内容结果如下图所示，此时还没有执行任何插入操作：

```text
db1=# BEGIN;
BEGIN
db1=# SELECT * FROM person;
 id | name
---+------
 1 | Alice
(1 row)
```

第二次查询 `person` 表内容结果如图所示，发现只插入了保存点 `s1` 前的新数据，说明事务回滚到保存点 `s1`：

> 注：这里的关键点是 `ROLLBACK TO SAVEPOINT s1` 只撤销 `s1` 之后的操作，所以 `InsertedBeforeS1` 会保留，而 `InsertedAfterS1` 会被回滚掉。
